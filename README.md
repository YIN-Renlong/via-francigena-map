## 📜 Abstract & Philosophical Foundation

This project is a custom-built, open-source publishing engine designed to document a 1,000-kilometer bicycle expedition from Rome to Bergamo along the Via Francigena, undertaken in August 2017. However, the architecture of the website is fundamentally an exercise in **Spatial Auto-Ethnography**. 

Viewed from the retrospective vantage point of nearly a decade later, the raw data of the 2017 journey (898 DSLR photographs, KML tracking files, and private diary entries) serves as a psychological baseline. It documents the operator's transition from a conflict-avoidant theology student navigating "immigrant politeness" in rural Italy, into a resilient, crisis-tested systems architect. 

To present this deeply personal and cultural evolution without relying on fragile, proprietary SaaS platforms (like ArcGIS StoryMaps) or ephemeral JavaScript frameworks (like React), this project embraces the philosophy of **Archival Stability**. By utilizing pure Vanilla HTML, CSS Grid, Vanilla JavaScript, and MapLibre GL JS, the codebase guarantees that this historical and emotional archive will remain legible, functional, and deeply interactive for decades to come.

## ⚙️ The Technical Architecture: A 3-Stage Pipeline

The website is not authored by hand; it is dynamically compiled by a custom Python pipeline that functions simultaneously as a Data Engineer, an Art Director, and a Typesetter. 

### 1. `process_day.py` (Somatic Data Harvesting)
This script acts as the foundational database builder. It parses raw `.kml` GPS files to extract somatic and environmental telemetry (altitude gradients, moving time, ambient temperature, kinetic energy expended). Concurrently, it utilizes Azure OpenAI Vision models to scan hundreds of original photographs, generating precise visual metadata and punchy, 30-word geographic captions tailored specifically for interactive map markers.

### 2. `process_daily_prose.py` (The Ethnographic Observer)
This script introduces the "Observer Paradox." It reads the author's highly private, vulnerable `memory.md` diary. Instructed to act as a Digital Humanities scholar, the Azure OpenAI LLM synthesizes the psychological reality of the trip (e.g., the "5% battery override," the anxiety of searching for shelter, the thermodynamics of empathy) with the visual metadata. It outputs a public-facing, structurally rigorous ethnographic essay, allowing the AI to act as a neutral, external lens decoding the author's internal Kantian ethics.

### 3. `build_website.py` (The Intelligent Typesetter)
The compiler. This script parses the AI's Markdown output and translates it into a dynamic web interface. It searches for custom "Layout Grammar" tags generated by the AI and converts them into advanced CSS/HTML wrappers. It also executes a mathematical interleaving loop, slicing the long-form essays and injecting interactive Map Cards between paragraphs to establish a specific editorial heartbeat. Finally, it outputs the static `config.js` file that drives the frontend.

## 🧭 UX & UI Philosophy: Translating Emotion to Interface

The user interface was designed to reflect the tension between physical exhaustion and intellectual reflection.

*   **The 3-Layer Asymmetric Dossier:** To bridge the gap between geographic reality and ethnographic reflection, the reading layout utilizes an advanced CSS pseudo-element (`::before`) to create an asymmetric, dark plum frame. The `#f9f6ed` reading layer floats inside it, leaving the 3D map fully visible and interactive in the margins. It turns the text into a physical "dossier" hovering over the earth.
*   **Dynamic Image Deduplication Engine:** A sophisticated Python Regex parser tracks every photograph the AI selects for the main essay. It appends these to a `used_images` array and actively strips them from the Map Slider queue, ensuring absolute visual uniqueness and preventing UI repetition.
*   **Smart Telemetry Dashboard:** A bottom-anchored data dashboard tracks speed, elevation, and weather. To respect "Reading Mode," Vanilla JS event listeners automatically hide the dashboard off-screen during text-heavy prose sections, sliding it back up only when the user returns to the geographic map cards.
*   **Cinematic Day Transitions:** Automated full-screen, dark-overlay cover images with hardcoded, elegant typography (Georgia and Open Sans) signal the transition between grueling days, preventing cognitive overload.

## 🛠️ The Interactive History: A Chronicle of Human-AI Co-Creation

The final architecture of this engine is the result of an intense, iterative dialogue between the human architect and the AI system. Below is a record of the primary developmental friction points and breakthroughs:

### Phase 1: Rejecting the Walled Garden & Establishing Rhythm
The initial challenge was escaping Mapbox/MapLibre's default template without falling into the "React trap." We committed to building a custom component renderer (`story-layouts.js`) that injects HTML directly into Scrollama.js.
We quickly realized that endless scrolling map cards caused "framework fatigue." We solved this by programming the Python compiler to chop the AI's essay into chunks using `##` headers, mathematically interleaving text blocks with 1-3 interactive map cards. This created a rhythm: *Read a thematic section -> Scroll through spatial geography -> Read the next reflection.*

### Phase 2: Inventing the "Layout Grammar"
To achieve high-end magazine layouts (reminiscent of *The New York Times* or *National Geographic*) using only Markdown, we taught the Azure OpenAI model to act as an Art Director. Instead of standard image tags, the AI was prompted to output specific, bespoke layout syntax:
*   `![FULL_BLEED](img.jpg)` -> Translated by Python into a `100vw` cinematic visual break spanning the absolute edges of the monitor.
*   `![DIPTYCH](img1.jpg | img2.jpg)` -> Translated into a perfect `1fr 1fr` CSS Grid comparison.
*   `![STICKY_RIGHT](img.jpg)` -> Translated into a floating portrait crop, allowing text to wrap elegantly around the image.

### Phase 3: The API Limit Crash & The "Reasoning" Era
During development, the pipeline suffered catastrophic `JSONDecodeError` crashes during Pass 2 narrative generation. Deep log analysis revealed that the cutting-edge Azure OpenAI models (`gpt-4.5-preview` / `o1` series) had deprecated the `max_tokens` parameter in favor of `max_completion_tokens`. 
Furthermore, because these are advanced "reasoning" models with a hidden internal "thinking phase," the model was exhausting its 4,000-token limit simply *thinking* about the 30+ photo metadata objects, outputting a blank string and crashing the JSON parser. We re-engineered the payload requests to support massive 15,000-token completion windows, successfully bypassing the architectural bottleneck.

### Phase 4: Extreme UX Polish & The Disappearance of the Interface
In the final iterations, we ruthlessly eliminated UI friction to center the cultural narrative:
1.  **The Progress Bar:** We deleted traditional text navigation links, which cluttered the screen, replacing them with a sleek, 4px reading progress bar calculated dynamically via Vanilla JS `window.scrollY` math.
2.  **The Lightbox:** We built a native, dark-mode Lightbox overlay to allow high-resolution image viewing without the jarring UX failure of opening new browser tabs.
3.  **Elegant Metadata:** We abandoned playful, "pill-shaped" tags for severe, minimalist CSS top-borders for GPS coordinates, honoring the serious, documentary tone of the project.

## 🖥️ Operational Guide: Compiling a New Day

This engine is designed to be fully automated once the raw data is provided. To process and publish a new day of the expedition (for example, August 9th, 2017), follow this exact workflow.

### The Standard Compilation Pipeline

**1. Prepare the Raw Data**
* Create a new folder for the day: `images/2017-08-09/`
* Drop all of the day's high-resolution `.jpg` photographs into this folder.
* Create a `memory.md` file inside the folder and write the private, unedited diary entry for that day.
* Ensure the corresponding GPS tracking file (e.g., `Cyclemeter-Cycle-20170809.kml`) is located in the `kml_raw/` folder.

**2. Extract Telemetry & Visual Metadata (Database Generation)**
Run the initial processing script in your terminal:

```bash
python3 process_day.py 2017-08-09
```

*What this does:* It parses the KML for GPS coordinates, elevation, and weather. It then uses Azure OpenAI Vision to scan every photo, generating `[filename].json` metadata and short 30-word map captions (`global_narratives.json`).

**3. Synthesize the Ethnographic Essay (Art Direction)**
Run the prose generator:

```bash
python3 process_daily_prose.py 2017-08-09
```

*What this does:* The AI acts as the Ethnographer and Art Director. It reads the private `memory.md`, views the available photo metadata, and writes a public-facing essay (`output_metadata/2017-08-09/generated_prose.md`). It automatically decides which photos to embed using specific layout grammar.

**4. Compile the Web Engine (The Typesetter)**
Run the final build script:

```bash
python3 build_website.py
```

*What this does:* It translates the Markdown layout tags into advanced CSS Grid/Flexbox HTML, filters out duplicate photos from the map slider queue, and stitches all data into the static `config.js` file.

**5. View the Output**
Open `index.html` in any modern web browser to view the finalized, interactive documentary.

### 🎨 System Overrides & Editorial Tips

The engine is built to allow the human Architect to override AI decisions when strict editorial control is required.

**The Cover Image Override (`cover.jpg`)**
By default, the compiler uses the chronologically first photograph of the day as the cinematic Full-Screen Cover. If you want a different photograph to act as the cover:
1. Duplicate your preferred photo inside the `images/YYYY-MM-DD/` folder.
2. Rename it exactly to `cover.jpg`.
3. Re-run `python3 build_website.py`. The compiler will automatically detect `cover.jpg` and use it for the day's transition screen.

**Updating the Timeline Titles**
To change the cinematic titles displayed over the cover images (e.g., "Day 3 OF 16" and "Bureaucracy and Grace"), open `build_website.py`. At the top of the file, locate the `journal_headers` dictionary and add/edit the specific date's metadata.

**Adding Photos to an Already Processed Day**
If you decide to add more `.jpg` files to a day that has already been compiled:
1. Put the new photos into the day's `images/` folder.
2. Go to `output_metadata/YYYY-MM-DD/` and **delete** `global_narratives.json`.
3. Re-run steps 2, 3, and 4. (Deleting the old narrative forces the AI to acknowledge the new photos and rewrite the map captions).

**Manual Markdown Layout Tags**
If you wish to manually edit the AI's essay (`output_metadata/YYYY-MM-DD/generated_prose.md`), you can force advanced CSS layouts by typing these specific tags:
* `![FULL_BLEED](_MG_1234.jpg)` : Breaks the reading container for a 100vw cinematic image.
* `![DIPTYCH](_MG_1234.jpg | _MG_5678.jpg)` : Places two photos in a perfect 50/50 CSS Grid.
* `![STICKY_RIGHT](_MG_1234.jpg)` : Auto-crops a landscape photo to a 4:5 vertical portrait and floats it on the right side of the text.
